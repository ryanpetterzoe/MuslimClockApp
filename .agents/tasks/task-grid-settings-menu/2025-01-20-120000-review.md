# Grid-based settings landing page replacing flat PreferenceScreen

The monolithic `SettingsFragment` (a single `PreferenceFragmentCompat` loading one massive `preferences.xml`) has been split into a grid landing page with nine category cards and eight sub-fragments, each owning its own preference XML and `ActivityResultLauncher`s. The ninth card ("Editor Tata Letak") opens `LayoutEditorActivity` directly. Layout-editor slider preferences (`*_size`, `*_x_pct`, `*_y_pct`) are intentionally removed from the settings UI since they now live exclusively in the live-preview editor.

**Watch for:** Double-call to `LogoStorage.clear()` in `MasjidSettingsFragment.confirmClearLogo` — the logo is deleted before the condition check, so the toast always reports "nothing to clear" (confirmed). Hardcoded dark-theme colors and pixel margins in the grid layout make the UI brittle on non-TV form factors (confirmed). The old `preferences.xml` is left on disk unreferenced — dead file that will confuse future contributors (confirmed).

## High-level view

The navigation model moves from a scroll-to-category pattern to a hub-and-spoke pattern. `SettingsActivity` keeps a back-stack listener that resets the title when the grid reappears, and each sub-fragment is a self-contained `PreferenceFragmentCompat` that registers its own file-picker launchers, so lifecycle concerns are properly scoped.

The preference XML split covers all 86 original keys — 65 moved into the 8 new files, 21 intentionally removed (layout editor controls now in `LayoutEditorActivity`, plus `copyright` moved to the grid footer).

The grid is built imperatively: cards are inflated in a loop and added to a `GridLayout` with computed `LayoutParams`. This avoids a RecyclerView adapter for a fixed 9-item grid, but it uses hardcoded pixel margins and color values that won't adapt to system theme changes or varying screen densities.

4. **Pixel-based margins ignore density** — Card margins (`8` in `GridLayout.LayoutParams`) are set in pixels via code, not dp. On high-density screens the spacing will be nearly invisible. Fix: `(8 * resources.displayMetrics.density).toInt()`.

## Double-delete bug in logo clearing

`MasjidSettingsFragment.confirmClearLogo()` calls `LogoStorage.clear(ctx)` on line 190, which deletes the file and returns a boolean. But the toast message is derived from a *second* call to `LogoStorage.clear(ctx)` on line 196:

```kotlin
LogoStorage.clear(ctx)                    // first call — deletes the file
val pref = findPreference<EditTextPreference>(Settings.K_MASJID_LOGO)
if (pref != null && LogoStorage.isLocalLogoUrl(pref.text)) {
    pref.text = ""
}
Toast.makeText(
    ctx,
    if (LogoStorage.clear(ctx)) R.string.logo_cleared   // second call — always false
    else R.string.logo_nothing_to_clear,
    Toast.LENGTH_SHORT
).show()
```

The original code stored the boolean from a single call (`val removed = LogoStorage.clear(ctx)`) and used it in the toast. The refactored version broke this by splitting the call into two invocations. The user always sees "nothing to clear" even when a logo was successfully deleted.

## Hardcoded dark styling

The grid uses hardcoded dark colors:

- `fragment_settings_grid.xml`: `android:background="#FF1A1A1A"`
- `bg_settings_card.xml`: `<solid android:color="#FF2D2D2D" />`
- `item_settings_card.xml`: `android:textColor="#FFFFFFFF"`

For a TV-only app this is workable, but if the app is ever run on a tablet in light mode or the system forces light theme, the settings screen will be invisible white-on-white or dark-on-dark. Extracting to `colors.xml` or theme attributes is low-cost future-proofing.

## Programmatic pixel margins

In `SettingsGridFragment.onViewCreated`, card layout params are built with:

```kotlin
params.setMargins(8, 8, 8, 8)
```

`GridLayout.LayoutParams.setMargins` takes pixels, not dp. On an xxhdpi screen (3x density) this is effectively ~2.7dp of margin. The XML layouts correctly use dp units; only this programmatic path is affected.

## Dead `preferences.xml`

The original monolithic `res/xml/preferences.xml` is still present on disk. No code references it anymore — the new fragments each load their own `pref_*.xml`. The task findings note it was "left in place as a backup," but in a shipped APK it's dead weight and a source of confusion.
