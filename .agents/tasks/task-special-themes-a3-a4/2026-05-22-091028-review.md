# Four standalone special theme layouts (A3/A4 x vertical/horizontal)

Four new self-contained HTML prayer clock pages, each ~470-510 lines. A3 variants use a glass-morphism card style (rounded corners, backdrop-blur). A4 variants use a bold geometric style (sharp 2px border-radius, solid opaque backgrounds, table-based prayer display). Vertical variants stack content in a column; horizontal variants use a left/right panel split. All four share the same JavaScript core: MCAndroid bridge config, Aladhan prayer time fetch, digital clock, countdown, slideshow with frame overlay, adzan/iqamah overlay, and ticker bar.

Watch for: No Android-side integration yet -- these files exist as assets but nothing loads them (confirmed). Massive code duplication across all four files means bugs must be fixed in four places (confirmed). The `masjid_logo` field is interpolated directly into an `innerHTML` assignment without sanitization, creating an XSS surface when config is attacker-controlled (confirmed).

## High-level view

The four files are purely additive web assets. The Android app currently only loads `index.html`; no Kotlin code references the new filenames. Until a theme-switcher or activity change is wired up, these pages are inert. This is likely intentional scaffolding, but it means the full feature isn't shippable from this change alone.

Each file duplicates the entire JavaScript runtime (~180 lines) with no shared module. Any future fix (e.g., fixing the Aladhan URL construction, adding error recovery, patching the XSS vector) needs to be applied four times. Extracting the common logic into a shared `.js` file would reduce maintenance cost significantly.

The Aladhan API call fetches an entire month's calendar to extract a single day's timings. If the device lacks internet or the API is unreachable, the pages show "--:--" forever with no retry and no cached fallback. The hourly `setInterval` refresh won't help if the network recovers faster than that.

The adzan overlay timer has a race condition: if the page loads within 0-59 seconds after a prayer time, the overlay fires immediately. If it loads at second 60 or later, that prayer is silently missed until the next day.

## XSS through logo config injection

In all four files, the `applyConfig` function contains:

```javascript
if(cfg.masjid_logo){
    const box = document.getElementById('logoBox');
    if(box) box.innerHTML = '<img src="'+cfg.masjid_logo+'" ...>';
}
```

If `masjid_logo` is set to `" onerror="alert(1)` or contains a closing tag followed by a script, the page executes arbitrary code. The `window.applyConfig` function is globally exposed and can be called from any injected script or URL handler. The fix is straightforward: create the `img` element via `document.createElement('img')` and set `.src` programmatically.

## Duplicated runtime with no shared module

The JavaScript IIFE in each file is functionally identical. The only variance is cosmetic (CSS classes like `glass-card` vs `geo-card`). The entire prayer-time fetch, clock update, adzan/iqamah state machine, slideshow controller, and config application logic is copy-pasted. This means:

- A bug fix must be applied to 4 files.
- Behavioral divergence will creep in as files are edited independently.
- The total asset size is inflated by ~4x what's necessary for the JS portion.

A single `spesial-clock.js` that the four HTML files `<script src>` would eliminate this. The HTML files would only contain their unique CSS and DOM structure.

## Prayer time fetch has no resilience

`fetchPrayerTimes` fires once at init and then every 3,600,000ms. If the first fetch fails (no network, API down, malformed response), `times` stays empty and the UI shows placeholders indefinitely. There's no exponential backoff, no shorter retry on failure, and no mechanism to receive cached times from the Android host.

The Aladhan URL fetches the full month's calendar (`/v1/calendar/{month}/{year}`) when only today's data is needed. The `/v1/timings/{timestamp}` endpoint would be lighter and avoids the array-index lookup that breaks if the device clock is wrong (day-of-month off by one from server perspective).

## Adzan trigger window and post-Isha countdown gap

The adzan check runs every second and fires if `nowSec - targetSec` is between 0 and 59 inclusive. This means:

- If the page is loaded or becomes visible more than 60 seconds after a prayer time, that prayer's adzan is permanently skipped for the day.
- After Isha passes, no `nextKey` is found, so the countdown shows "Subuh --:--:--" instead of computing the actual hours until tomorrow's Fajr. This is a visible UI gap during the nighttime hours between Isha and midnight (and midnight to Fajr).
