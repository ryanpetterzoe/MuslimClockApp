# Theme pruning, new layouts, and layout editor expansion

This change removes ~30 unused or unsupported layout themes from the codebase (SUPPORTED_LAYOUTS, arrays.xml, and HTML templates), adds 8 new photo/fullphoto templates with their prayer-highlight CSS, fixes photo-frame opacity on 4 themes that had hardcoded dark backgrounds competing with the user's `--slide-opacity` setting, makes the adzan overlay text responsive via `clamp()` to prevent clipping when imam data is shown, and expands the layout editor with dropdown/toggle controls for digital style, quran mode, identity position, and logo position. The default layout changes from `minimal` (now removed) to `cinema`.

Watch for:
- **Orphaned CSS for damascus and imperial** (confirmed) — their CSS card/highlight rules remain in screen.css but they have no template, no SUPPORTED_LAYOUTS entry, and no arrays.xml entry. Dead code that will never execute.
- **Logo position DOM manipulation fragility** (likely) — the "top" logo position logic uses `identParent.tagName !== 'HEADER'` as guard, but several templates place `#logoBox` directly inside the `<header>` flex container, meaning the guard prevents the feature from working on those templates.
- **arrays.xml pruning far exceeds the described scope** (confirmed) — 30+ themes removed from arrays.xml/SUPPORTED_LAYOUTS beyond the 9 ornamental+dome themes mentioned in the task description. Not a bug, but the PR description undersells the blast radius.

## High-level view

The layout catalog was pruned from ~95 entries to 64, removing themes that never had working templates or were deemed redundant. The sync between SUPPORTED_LAYOUTS in clock.js and arrays.xml layout_values is correct — both carry exactly 64 entries in matching order. The fallback layout changed from `minimal` to `cinema` across all three touch points (clock.js DEFAULTS, Settings.kt, pref_appearance.xml). Any existing user who had a removed theme selected falls through to `cinema` via the guard in `mountLayout()`.

Eight new templates (oasis, arch, courtyard, panorama, fullphoto4, fullphoto5, fullphoto12, fullphoto18) follow the established contract with all required IDs. Each gets a corresponding `.prayer.next` CSS rule with glow/scale effects and `::before` suppression.

The adzan overlay converts fixed Tailwind sizes to `clamp()` expressions scaled by viewport height, with `max-h-screen overflow-y-auto` as safety net. On Android TV with D-pad navigation, if overflow does occur, there's no way for the user to scroll.

The layout editor adds dropdowns (digital style, identity position, logo position, quran mode) and a toggle (hide seconds). The "Logo" element was consolidated under the "Identitas" tab. New identity X/Y positioning translates the entire `<header>`, which couples date movement with identity movement.

## Orphaned CSS for damascus and imperial

`.layout-damascus .dm-card`, `.layout-damascus .prayer.next .dm-card`, `.layout-imperial .im-card`, `.layout-imperial .prayer.next .im-card` — roughly 50 lines of CSS card rules remain in screen.css. The `::before` suppression rule at ~line 1290 also still references both:

```css
.layout-lantern   .prayer.next::before,
.layout-damascus  .prayer.next::before,
.layout-imperial  .prayer.next::before {
    display: none;
}
```

No template with class `.layout-damascus` or `.layout-imperial` exists anywhere in the codebase. These were already orphaned in the base branch (no SUPPORTED_LAYOUTS entry, no template) but had arrays.xml entries. This PR removed the arrays.xml entries but missed the CSS cleanup.

## Adzan overlay responsive sizing on small displays

On a 1080p TV, the countdown `clamp(3rem, 12vh, 8rem)` hits the 8rem cap (12vh = 129.6px > 128px). On a 720p display, `12vh` = 86px which is comfortably within range. The risk scenario is a 720p display with imam name + khatib name + full quran card + countdown all visible simultaneously — that's roughly 6-7 text blocks stacking vertically. If total content exceeds viewport, `overflow-y-auto` kicks in but the D-pad on Android TV can't scroll it.

## Logo position DOM manipulation

```javascript
const identParent = box.parentElement;
if (identParent && identParent.tagName !== 'HEADER') {
    if (logoPos === 'top') {
        identParent.style.flexDirection = 'column';
        identParent.insertBefore(box, identParent.firstElementChild);
    } else {
        identParent.style.flexDirection = '';
    }
}
```

Most templates place `#logoBox` inside `<div class="flex items-center gap-3">` within `<header>`, so `identParent` is that wrapper div and the feature works. Templates where `#logoBox` is a direct child of `<header>` will silently ignore the logo position setting. An exhaustive audit of all 64 templates would clarify how many are affected.

## Identity X/Y translates the full header

```javascript
const identityHeader = (function () {
    const el = document.getElementById('logoBox') || document.getElementById('masjidName');
    return el ? el.closest('header') : null;
})();
if (identityHeader) {
    identityHeader.style.transform = isDefault ? '' : `translate(${ix}vw, ${iy}vh)`;
}
```

This moves the entire header including the date block. A user adjusting "Identitas" position will also shift their date display. If the intent is to move only the identity block, the transform target should be the logo+name wrapper div rather than the full header.

## Scope of theme pruning

Beyond the 9 ornamental+dome themes described, these were also removed from SUPPORTED_LAYOUTS and arrays.xml: minimal, mosque, neon, aurora, frame, stadium, magazine, split, portrait, kinetic, terminal, brutalist, sunrise, calligraphy, ottoman, celestial, rumi, andalusia, medina, batik, journal, canvas, bulletin, special2. The HTML templates for the ornamental 9 are removed in this diff; the others were likely removed in prior work on the base branch or never had templates. The correctness constraint — SUPPORTED_LAYOUTS matches arrays.xml layout_values — holds.
