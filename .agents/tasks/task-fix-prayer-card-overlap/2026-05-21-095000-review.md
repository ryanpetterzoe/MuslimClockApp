# Vertical overflow fix for prayer card layouts

The change prevents prayer schedule sections from overlapping other content in short/landscape viewports across all 80+ layout templates in the WebView clock app. It takes a three-layer approach: CSS flex-shrink allows row-fixed elements to compress naturally, a media query strips padding on very short viewports, and a new JS function (`autoFitVerticalOverflow`) detects when row-fixed items consume more than 60% of the container and applies a CSS `scale()` transform to prayer sections. The fix is global — it targets `.row-fixed` and dynamically queries the active layout at runtime, so no per-template modifications were needed.

**Watch for:** The scale transform does not reduce the element's layout box, so the `row-fill` main area doesn't reclaim space when sections are scaled down (confirmed). The `!important` padding override in the media query suppresses intentional padding from all templates (including headers) at ≤500px (likely). No debounce on the vertical overflow resize handler causes redundant executions and potential flicker (confirmed).

## High-level view

The CSS change from `flex: 0 0 auto` to `flex: 0 1 auto` is the most impactful single line — it lets the flex container shrink row-fixed children when total content exceeds the viewport. Adding `overflow: hidden` prevents text from spilling out of a shrunken container.

The JS `autoFitVerticalOverflow` function is a second line of defense for cases where flex-shrink alone isn't enough. It applies a CSS `scale()` transform to prayer sections, but because transforms don't participate in flex layout, the scaled element still occupies its full pre-transform height. The function skips elements that transparent passthrough has taken out of flow, but the timing between the two systems is implicit (50ms setTimeout) rather than explicitly sequenced.

The media query at `max-height: 500px` collapses padding on *all* `.row-fixed` elements with `!important`, which will override template-specific padding on headers (mosque name, identity logos) — not just prayer sections.

<details>
<summary>Issues (4)</summary>

1. **Scale transform doesn't shrink layout box** — CSS `transform: scale()` does not change layout dimensions. A section scaled to 65% still occupies 100% of its original height in the flex algorithm. The `row-fill` main area (clock) doesn't gain space back. Templates with fixed-height card backgrounds or images would hit this limitation hardest.

2. **`!important` padding override is indiscriminate** — The `@media (max-height: 500px)` rule forces 2px padding on *all* `.row-fixed` elements, including headers with mosque identity that need padding for legibility. Scope to `section.row-fixed` only, or use a less aggressive value.

3. **No debounce on vertical overflow resize handler** — The existing `autoFitPrayerCards` uses a proper debounce (clearTimeout pattern). The new vertical handler uses raw `setTimeout(autoFitVerticalOverflow, 150)` without canceling previous timers, so rapid resize events queue multiple executions. Each resets transforms and triggers layout flushes, causing flicker. Use the same clearTimeout pattern.

4. **Passthrough timing is implicit** — `autoFitVerticalOverflow` runs via `setTimeout(..., 50)` after synchronous `applyTransparentPassthrough()`. The rAF callback inside does re-check computed positions, so it currently works. But if passthrough ever becomes async or a config update resets positions mid-timer, measurements will be wrong. An explicit sequencing guarantee (e.g., calling overflow detection from passthrough's completion) would be more robust.

## Scale transform as a visual-only fix

```javascript
s.el.style.transform = 'scale(' + scaleFactor + ')';
s.el.style.transformOrigin = 'center bottom';
```

CSS transforms don't participate in flex layout. A scaled-down section still occupies its full pre-transform height in the flex algorithm:

- `flex: 0 1 auto` does the real work of preventing overflow at the layout level.
- The scale transform is a cosmetic enhancement for cases where flex-shrink reaches a floor (element has large min-content height, e.g. a prayer card grid with fixed-height children). In those cases, the visual scale hides the overflow but the `row-fill` main area (clock) doesn't gain that space back.

If the goal is to actually reclaim space, the function would need to set an explicit `height` on the section (to `scaleFactor * naturalHeight`) alongside the transform, or use a different approach entirely (font-size reduction, which does affect layout).

## Resize handler accumulation

The existing pattern for horizontal autofit:

```javascript
let _autoFitTimer = null;
const scheduleAutoFit = () => {
    if (_autoFitTimer) clearTimeout(_autoFitTimer);
    _autoFitTimer = setTimeout(() => { _autoFitTimer = null; autoFitPrayerCards(); }, 120);
};
```

The new vertical handler:

```javascript
window.addEventListener('resize', function() { setTimeout(autoFitVerticalOverflow, 150); });
```

Each resize event creates a new 150ms timer without canceling the previous one. During a window resize drag, this fires the function dozens of times in succession. Each invocation resets transforms (triggering a layout flush), waits one rAF, then re-applies — causing visible flickering of the prayer section scaling up and down. The fix is the same clearTimeout pattern already used 15 lines above.

## Media query scope

```css
@media (max-height: 500px) {
    .flex-layout > .row-fixed { padding-top: 2px !important; padding-bottom: 2px !important; }
}
```

500px viewport height is a common landscape phone resolution. At that size, headers containing mosque identity (name, logo, location text) will have their padding crushed to 2px. Scoping to `.flex-layout > section.row-fixed` would preserve header breathing room while still addressing the prayer card overflow.
